# AGENTS.md

## Назначение проекта

Noon Tracker — Telegram-бот, отслеживающий офферы товаров на `https://www.noon.com/uae-en/`.

Пользователь передает ссылку на товар и опциональную целевую цену. Бот должен уведомить пользователя в двух случаях:

1. Изменился лидер — продавец с самым дешевым доступным оффером.
2. Появился оффер с ценой, равной целевой цене пользователя или ниже нее.

Данные по активным товарам должны обновляться каждую минуту.

## Технологический контекст

- Framework: NestJS
- Telegram: Telegraf
- Jobs: BullMQ
- Queue storage: Redis
- Database: MongoDB через Mongoose
- Language: TypeScript
- Package manager: pnpm
- Runtime/deployment: Docker Compose

## Границы модулей

- `src/bot` отвечает только за Telegram-интерфейс и преобразование пользовательского ввода в вызовы прикладных сервисов.
- `src/offers` отвечает за ссылки Noon, получение, нормализацию и сравнение офферов.
- `src/queues` отвечает за планирование проверок, обработчики заданий и retries.
- Инфраструктурные подключения должны оставаться в `src/infrastructure` или в специализированных инфраструктурных модулях.
- Контроллеры должны быть тонкими; бизнес-правила размещаются в сервисах.

## Доменные правила

- Канонический идентификатор товара — SKU из URL Noon, а не полный URL.
- Tracking/query-параметры URL не должны создавать отдельные товары.
- Разные варианты товара с разными SKU считаются разными товарами.
- Один SKU запрашивается только один раз за минутный цикл, даже если на него подписано несколько пользователей.
- Лидер определяется только среди доступных офферов по минимальной базовой цене.
- Для сравнения продавцов использовать устойчивый идентификатор продавца/оффера. Имя продавца не является надежным идентификатором.
- Денежные значения нельзя хранить и сравнивать через неточные вычисления с floating point. Использовать целое количество минимальных денежных единиц либо Decimal128.
- Регион и контекст доставки должны быть явными и одинаковыми между проверками.
- Карточные скидки, промокоды, cashback и персональные акции не входят в базовую цену оффера без отдельного требования.

## Состояние подписки

Подписка должна как минимум связывать:

- Telegram user/chat;
- товар/SKU;
- опциональную целевую цену;
- статус активности;
- состояние срабатывания целевой цены.

После уведомления о достижении цены не отправлять его каждую минуту. Событие можно перевооружить, когда цена снова станет выше целевой, а затем повторно пересечет порог вниз.

## Проверка и события

Рекомендуемый поток:

1. Планировщик ставит задание проверки уникального активного SKU.
2. Worker получает и нормализует офферы Noon.
3. Результат сравнивается с последним успешно сохраненным состоянием.
4. Изменение лидера и пересечение ценового порога создают доменные события.
5. Состояние и события сохраняются атомарно либо с идемпотентными ключами.
6. Отдельный worker отправляет Telegram-уведомления.

Retries не должны приводить к повторным уведомлениям. Для job/event/notification использовать устойчивые идентификаторы и уникальные ограничения там, где это возможно.

## Работа с Noon

- Не привязывать доменную логику напрямую к HTML-селекторам или внешнему JSON.
- Ответ Noon сначала преобразовывать во внутренний DTO/модель оффера.
- Считать внешний формат нестабильным и валидировать обязательные поля во время выполнения.
- Устанавливать таймауты на все сетевые запросы.
- Использовать ограниченный exponential backoff с jitter.
- Отличать отсутствие офферов от ошибки получения данных; ошибка парсинга не должна выглядеть как исчезновение всех продавцов.
- Логировать SKU, attempt и тип ошибки, не записывая секреты и персональные данные.
- Учитывать rate limiting и anti-bot защиту. Не увеличивать частоту запросов без необходимости.

## Telegram

- Валидировать, что ссылка относится к разрешенному домену Noon и UAE-разделу.
- Давать пользователю понятные ответы на неверную ссылку, цену или недоступный товар.
- Запускать bot polling/webhook только после регистрации handlers.
- Корректно останавливать Telegraf при shutdown приложения.
- Не проглатывать ошибки Telegram API без логирования.

## Конфигурация и безопасность

- Не коммитить `.env`, Telegram token, proxy credentials или другие секреты.
- Все адреса сервисов, расписания, таймауты и параметры retries задавать через конфигурацию.
- Валидировать обязательные переменные окружения при старте приложения.
- Значение по умолчанию для Noon должно указывать на `https://www.noon.com/uae-en`.
- Производственный интервал проверки — одна минута, если задача не требует другого.

## Требования к изменениям

- Сохранять строгую типизацию и не использовать `any` без обоснованной необходимости.
- Не смешивать получение внешних данных, сравнение состояния и отправку уведомлений в одном классе.
- Новые бизнес-правила покрывать unit-тестами.
- Парсер проверять на сохраненных обезличенных fixtures, чтобы тесты не зависели от доступности Noon.
- Интеграции с MongoDB, Redis и очередями покрывать интеграционными тестами для критичных сценариев.
- Перед завершением задачи запускать как минимум `pnpm run build` и релевантные тесты.
- Не исправлять посторонние изменения в рабочем дереве без явного запроса.

## Критичные сценарии тестирования

- Несколько пользователей подписаны на один SKU, но выполняется один запрос к Noon.
- Лидер изменился при одинаковой цене.
- Цена изменилась, но лидер остался прежним.
- Цена впервые пересекла пользовательский порог вниз.
- Цена остается ниже порога несколько циклов без повторного уведомления.
- Цена поднялась выше порога и затем снова опустилась.
- У товара временно нет доступных офферов.
- Noon вернул ошибку, rate limit или несовместимый формат.
- Worker повторно выполняет то же задание после сбоя.
- Telegram API временно недоступен и отправка повторяется без дубликатов.

## Текущий статус репозитория

Проект находится на стадии каркаса. Наличие файла или модуля не означает, что соответствующий сценарий уже реализован. Перед новой задачей сначала проверять фактическую сборку, DI-граф, регистрацию BullMQ queues/processors и запуск Telegraf.
