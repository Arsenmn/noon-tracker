# Рабочий процесс Noon Tracker

Связанные документы: [архитектура](architecture.md), [API Noon](api.md), [база данных](database.md), [очереди](queues.md).

## 1. Запуск приложения

NestJS валидирует переменные окружения, подключается к MongoDB и Redis, регистрирует Telegram handlers и BullMQ processors. При завершении процесса Telegraf, очереди и инфраструктурные соединения закрываются через lifecycle hooks.

## 2. Добавление товара

1. Пользователь отправляет UAE-ссылку Noon, при необходимости добавляя цену: `<ссылка> 899.99`.
2. Бот проверяет домен, раздел URL и извлекает SKU.
3. `NoonClientService` получает текущие офферы.
4. Бот отдельным сообщением показывает лидера и остальные доступные предложения.
5. Если цена не была указана, бот предлагает ввести её или пропустить шаг.
6. Подписка сохраняется активной и отслеживается до явной остановки.

Целевая цена считается достигнутой при `offer.priceMinor <= targetPriceMinor`, а не только при точном равенстве.

## 3. Получение сессии и данных Noon

Перед Axios-запросом клиент вызывает `getSession(productUrl)`:

- при живом кеше возвращаются сохранённые cookies и User-Agent;
- при пустом или истёкшем кеше Playwright открывает страницу товара, пытается перехватить catalog request и извлекает новые значения;
- если catalog request не появился, используются cookies из browser context;
- несколько одновременных callers ожидают одно обновление;
- Playwright и Axios используют один proxy и одинаковый сетевой контекст;
- Axios отправляет `Cookie`, `User-Agent`, `Referer`, country, locale и zone code;
- anti-bot ответы `401`, `403`, `429` и `503` вызывают одно обновление сессии и повтор запроса.

Лог `getSession output` явно показывает, была ли использована browser/cache сессия, какие cookie names найдены и когда кеш истекает. Секретные значения cookie в лог не попадают.

## 4. Минутный цикл мониторинга

1. Cron `OFFERS_CRON` читает активные подписки.
2. Подписки группируются по SKU.
3. Для каждого уникального SKU создаётся один BullMQ job на текущую минуту.
4. `MonitoringProcessor` вызывает `monitorSku(sku)`.
5. Клиент получает и нормализует офферы Noon.
6. Успешный снимок записывается в `monitored_products`.
7. Один снимок сравнивается со всеми подписками на SKU.

Если Noon недоступен или формат ответа несовместим, сохранённый успешный снимок не заменяется фиктивным пустым списком. Job завершается ошибкой и повторяется согласно настройкам retries.

## 5. События сравнения

### Смена лидера

При первой успешной проверке лидер сохраняется без уведомления. Далее сравнивается устойчивый `offerId`. Если он изменился, создаётся событие `leader-changed`, даже когда старая и новая цены равны.

### Целевая цена

Для подписки с целевой ценой выбирается самый дешёвый доступный оффер не дороже порога. При первом пересечении бот уведомляет пользователя и устанавливает `targetPriceTriggered=true`. Следующие минутные циклы не дублируют сообщение. Когда все офферы снова становятся дороже порога, флаг сбрасывается; следующее пересечение вниз снова создаст событие.

## 6. Доставка уведомления

1. Publisher upsert-операцией создаёт `notification_events` с уникальным `eventId`.
2. Тот же ID используется как BullMQ `jobId` в `telegram-notifications`.
3. `NotificationProcessor` атомарно переводит событие в `sending`.
4. Telegram-сообщение отправляется отдельно от monitoring worker.
5. Подписка продвигается, а событие получает статус `sent`.
6. Временная ошибка записывает `failed`; BullMQ повторяет отправку с exponential backoff.

## 7. Управление подписками

- `/list` показывает названия активных товаров и целевые цены;
- `/stop` выводит выбор отслеживаемых товаров и деактивирует выбранную подписку;
- `/cancel` отменяет незавершённый ввод;
- остановленный товар больше не попадает в минутный цикл, если на SKU не осталось других активных подписчиков.

## 8. Проверка разработчиком

```bash
pnpm test
pnpm run build
docker compose up -d redis mongodb
pnpm run test:e2e
```

Для проверки реальной Noon-сессии запустите приложение и добавьте товар. В логах должны последовательно появиться `getSession called`, `getSession output` и `Sending Noon request`. После первого browser refresh следующие запросы до истечения TTL должны иметь `source=cache`.
